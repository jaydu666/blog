Bully算法(长者为大)

核心思想: 基于序号选举算法,长者为大

  1. Bully 算法在选举过程中，需要用到以下 3 种消息：

    1. Election 消息，用于发起选举；
    2. Alive 消息，对 Election 消息的应答；
    3. Victory 消息，竞选成功的主节点向其他节点发送的宣誓主权的消息。

  2. 选举条件:

    1. 初始化
    2. 主节点宕机

  3. 选举规则:
    Bully 算法选举的原则是“长者为大”，意味着它的假设条件是，集群中每个节点均知道其他节点的 ID。

    1. 集群中每个节点判断自己的 ID 是否为当前活着的节点中 ID最大的，如果是，则直接向其他节点发送 Victory 消息，宣誓自己的主权；
    2. 如果自己不是当前活着的节点中 ID 最大的，则向比自己 ID 大的所有节点发送 Election 消息，并等待其他节点的回复；
    3. 若在给定的时间范围内，本节点没有收到其他节点回复的 Alive 消息，则认为自己成为主节点，并向其他节点发送 Victory 消息，宣誓自己成为主节点；若接收到来自比自己 ID 大的节点的 Alive 消息，则等待其他节点发送 Victory 消息；
    4. 若本节点收到比自己 ID 小的节点发送的 Election 消息，则回复一个 Alive 消息，告知其他节点，我比你大，重新选举。

  4. 自己理解:

    1. 人物介绍
      2. 老大: 序号最大
      3. 其他: 非序号最大
      4. 大哥: 大于自身序号的序号
      5. 小弟: 小于自身序号的序号
    2. 选举过程
      2. 老大: 直接告诉其他小弟,我是老大(victory)
      3. 其他: 征求大哥意见(election)就行了,小弟就不用问了
      4. 其他: 大哥们这么久不答复我(alive),估计挂了,那自己就当大哥(victory)了。如果收到大哥消息(alive),就大哥们决出胜负(等待victory),不关我的事了。
      5. 大哥: 收到小弟要当大哥消息(election),回话给小弟不要痴心妄想(alive)

  5. 应用:

    1. MongoDB 的副本集故障转移功能。MongoDB 的分布式选举中，采用节点的最后操作时间戳来表示 ID，时间戳最新的节点其 ID 最大，也就是说时间戳最新的、活着的节点是主节点。

Raft 算法(民主投票)

核心思想: 少数服从多数

  1. 角色:

    1. Leader,即主节点,同一时刻只有一个 Leader,负责协调和管理其他节点。
    2. Candidate，即候选者，每一个节点都可以成为 Candidate,节点在该角色下才可以被选为新的 Leader。
    3. Follower,Leader 的跟随者，不可以发起选举。

  2. 时间

    1. 心跳间隔: 主节点隔多长时间发送心跳信息
    2. 等待时间(election timeout): 如果超过这个时间仍然没有收到心跳,则认为主节点宕机。一般每个节点各自在 150～300ms 间随机取值。

    心跳间隔时间 < 等待时间

  3. 选举周期term:

    1. 作用: 第几轮选举,逻辑上的"时间",用于记录和比较各节点"进度"
    2. 某个节点收到信息的term > 自身term, 更新 自身term=收到更大的term, 进入Follower状态
    3. 某个节点收到信息的term <= 自身term, 忽略
    4. 例如由于网络情况不佳，一个主节点 A 与其它节点失联，其它节点选了一个新的主节点 B，当网络恢复正常时，旧主节点 A 收到主节点 B 的消息时，它会判断新主节点 B 的 term 大于自己，说明自己错过了一些事件，因此选择放弃自己的主节点身份

  4. 选举流程:

    1. 初始化时,所有节点均为Follower节点
    2. Follower节点等待超时,转为Candidate(由于等待时间是随机取值,Follower同时超时小概率,但有可能同时多个Follower超时),并向其他节点发送选举请求,同时把自己那票投给自己。
    3. 其他节点根据收到选举请求顺序,回复是否同意成为主。为了保证只能选出一个主,每一个节点只能投出一张票,而且票数超过一半才能成为主。
    4. 若发起选举的请求节点获得超过一半的票数,才能成为主,转为leader,其他节点状态由Candidate转为Fallower。Leader节点与Fallower节点之间定期发送心跳包,检测主节点是否活着。

  5. 相关资料:

    1. https://lotabout.me/2019/Raft-Consensus-Algorithm/ 相关文章
    2. http://thesecretlivesofdata.com/raft/#election 选举演示

Zab (Zookeeper Atomic Broadcat)

  1. 选举参考物:
    1. epoch: 选举轮数
    2. server_id: 节点唯一ID
    3. server_zxId: 存放数据ID(数据版本号, 每次commit都递增)
  2. 选举规则:
    优先选出epoch大的, epoch相同再选server_zxId大的, server_zxId相同再选server_id大的
